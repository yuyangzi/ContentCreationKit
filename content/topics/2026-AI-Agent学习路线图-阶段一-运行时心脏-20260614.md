## 本阶段定位

**一句话**：读完能用原生 SDK 写一个可靠的 Function Calling Agent，并理解为什么它有时候会失败。

**读者最终产出**：一个约 100 行的 Python Mini Agent Loop——不依赖任何框架，从零实现 Observe→Plan→Execute→Observe 循环，挂载天气查询和文件读写两个工具。与总纲的翻车案例形成叙事弧：总纲让你看到 Agent 在黑盒里装死 → 阶段一让你亲手造出底层，从此不会被任何框架挡住。

**在路线图中的位置**：地基阶段。这是整个 Agent 学习路径中最关键的阶段——如果 Function Calling 没搞透，后面用 LangChain/LangGraph 都是在黑盒操作。

### 模块一：Agent 不可靠性的根因分析（15%）

**定位收窄：不讲 LLM 科普，只讲"Agent 为什么有时候不调工具"。**

| 知识点 | 为什么重要 | 深度 |
|--------|-----------|------|
| 推理 vs 训练 | Agent 运行在推理模式下，行为和训练数据没有直接关系 | 够用即可 |
| 为什么 LLM 输出不可靠 | 概率性生成 → 同样的 prompt 同样的模型，有时调工具有时不调 | 讲透 |
| Temperature 对 Tool Call 的影响 | Temperature 直接控制模型"调用意愿"的稳定性——这是理解 tool_choice 行为的关键变量 | 讲透 |
| Token 与上下文窗口 | 对话历史占用 token，影响 long-running Agent | 够用即可 |

**不讲的内容**（留给感兴趣的读者自行深入）：
- Transformer 架构细节（Attention、QKV）
- MoE（Mixture of Experts）原理
- 预训练 / SFT / RLHF 训练流程
- Embedding 原理（阶段二会涉及）
- Token 经济（并入模块二的对话历史基础）

### 模块二：上下文工程（20%）

**从"写 Prompt"升级到"设计上下文窗口"。**

| 知识点 | 核心内容 |
|--------|----------|
| System Prompt 架构 | 角色定义 + 能力声明 + 行为约束 + 输出格式，四层结构设计 |
| 工具描述工程 | Tool Schema 的 name/description 如何影响模型选择工具的准确性——一个好的 description 比调参数更有效 |
| 对话历史管理基础 | messages 数组的构造规则（system/user/assistant/tool 四角色）、工具结果如何正确追加 |
| Few-shot 在 Agent 中的应用 | 简单提一下——用带 tool_calls 的对话示例教模型正确调用工具 |

**不讲的内容**（留给阶段四）：
- 滑动窗口 vs 摘要压缩 vs 选择性保留（高级对话管理策略）
- 上下文预算控制与 token 优化

**关键原则**：System Prompt 是第一道防线。Agent 不出问题不是靠"写得更好"，而是靠"约束得更精确"。

### 模块三：Function Calling + Agent Loop（47%）——核心模块

**这是全篇的重头戏。先讲理想情况（一次一个 tool_call），代码叙事为"大揭秘"模式：模块一二铺理论 → 模块三一把抛出完整的 Mini Agent Loop 逐行讲解。**

> ⚠️ 2026 年 FC 新默认行为（`parallel_tool_calls` 默认开启、`strict: true` 结构化输出、`refusal` 字段、流式 tool calls）将在模块四 Harness Engineering 中作为"进阶坑"统一讲解。

#### 3.1 Function Calling 底层机制

```
用户消息 → LLM（含 tools 定义）
              ↓
         LLM 决策：回复文本 或 返回 tool_calls
              ↓
    tool_calls 包含：id, function.name, function.arguments (JSON)
              ↓
         执行工具函数，结果回传给 LLM
              ↓
         LLM 综合结果，生成最终回复
```

| 关键参数 | 作用 | 常见坑 |
|----------|------|--------|
| `tool_choice` | 控制是否/如何调用工具（auto/none/required/指定工具） | 设为 required 但工具定义有歧义 → LLM 幻觉一个调用 |
| `tools[].function.parameters` | JSON Schema 定义参数 | 类型不严谨 → LLM 传字符串而非数字 |
| `finish_reason` | 判断下一步：stop/tool_calls/length/content_filter | 不检查 finish_reason 直接读 content → 拿到 null |
| `tool_call_id` | 将工具结果绑定到正确的 tool call | ID 对不上 → LLM 忽略你的返回值 |

**Tool Schema 设计三条原则**（来自实战血泪教训）：
1. **明确性**：`description` 写清楚"什么时候用这个工具、输入输出是什么"
2. **最小权限**：不要让一个工具做太多事，拆分粒度影响调用成功率
3. **幂等性**：相同输入 → 相同输出，Agent 可能对同一条消息重复调用

#### 3.2 Agent Loop：观察 → 规划 → 执行 → 观察

```
┌──────────────────────────────────────────┐
│              Agent Loop                   │
│                                           │
│  Observe ──→ Plan ──→ Execute ──→ Observe │
│    ↑                                  │   │
│    └──────────── 循环 ────────────────┘   │
└──────────────────────────────────────────┘
```

| 步骤 | 做什么 | 代码实现 |
|------|--------|----------|
| Observe | 读取上一步的执行结果或用户输入 | `response = client.chat.completions.create(...)` |
| Plan | LLM 决定下一步：回复用户还是调用工具 | 检查 `finish_reason == "tool_calls"` |
| Execute | 实际执行工具函数 | `tool_result = tool_fn(**json.loads(tool_call.function.arguments))` |
| Loop | 将工具结果回传 LLM，继续循环 | 将结果以 `role: "tool"` 追加到 messages |

**循环终止条件**：
- `finish_reason == "stop"` → Agent 认为任务完成
- 达到最大循环次数（防止无限循环）
- 连续两次相同 tool call（检测死循环）

### 模块四：Harness Engineering 驾驭工程（15%）

**"Human Steer, Agent Execute"——给 Agent 套上缰绳。**

| 机制 | 实现方式 | 防止的问题 |
|------|----------|-----------|
| 约束边界 | 工具函数内做权限检查（如：禁止删除生产数据） | Agent 越权操作 |
| 反馈回路 | 工具返回结构化错误信息（success + error_message），Agent 据此自我纠正 | 静默失败 |
| Human-in-the-loop | 关键操作前中断 Agent Loop，等待人工确认 | 不可逆操作（付款、发邮件） |
| 安全护栏 | 输入过滤（防止 prompt injection）+ 输出过滤（防止敏感信息泄露） | 安全攻击 |
| 超时与重试 | 工具函数超时 + 指数退避重试 + 最终降级策略 | 工具调用卡死 |

## 技术细节

- **主教程代码**：使用 **OpenAI SDK**（兼容 DeepSeek API），原生调用展示底层机制
- **教程模型**：**`deepseek-v4-flash`**（非废弃的 `deepseek-chat` 别名）。配置：`base_url="https://api.deepseek.com"`，显式禁用思考模式 `"thinking": {"type": "disabled"}` 以避开 `tool_choice` 的 known issues，统一用 `tool_choice="auto"`
- **末尾对比**：用 LangChain `@tool` 装饰器写同样功能的 5 行代码版本——"你现在知道它在底层干了什么"
- **多模型切换**：标注替换 `base_url` + `api_key` + `model` 即可切到 OpenAI / Claude / Qwen 的说明

## 差异化价值

- **原生 SDK 教学**：不是 LangChain 包装后的黑盒，读者能看到 tool_choice / finish_reason / tool_call_id 的完整流转
- **Agent Loop 图景**：不把 Function Calling 当"一次性调用"，而是讲清楚循环决策的完整机制
- **Harness Engineering 概念**：2026 年前沿理念，给 Agent 套安全缰绳，不是只教"怎么调 API"
- **从坑出发**：每个知识点从"Agent 在这里会怎么坏"开始讲
