# Agent Loop工程：AI自己干活、监工和返工

## 热度背景

2026年6-7月，**Loop工程**（Agent循环工程）从一个边缘概念迅速成为AI产业最热工程范式。Claude Code创始人Boris Cherny最早提出，Google工程负责人Addy Osmani系统性整理，吴恩达在X上公开推荐，Claude Code官方博客全面公开四种Loop原语——一个月之内，Loop工程完成了从「没人提」到「行业共识」的转折。

与此同时，GitHub上6万Star的Ponytail插件、全球各Agent产品（Claude Code / Codex / ZCode / MiniMax Code / Workbuddy / Kimi Work）集体将「写—跑—看结果—再改」做成默认能力，信号非常明确：**Agent不再等你提示词，它在自己转了。**

### 核心概念

**Loop工程关心的是：如何让Agent围绕一个目标不断「循环」运作——理解任务→调取上下文→调用工具→观察结果→判断是否完成→没完成就继续改。**

Anthropic将Loop分成四种：

| 类型 | 含义 | 例子 |
|------|------|------|
| Turn-based（回合制） | 人发一条，AI回一条 | 常规对话 |
| Goal-based（目标驱动） | 给目标，AI自己写到通过 | 「帮我写个登录页面」 |
| Time-based（定时触发） | 按时间自动执行 | 每2小时检查PR更新并review |
| Proactive（主动式） | AI自己发现问题自己干 | 发现测试覆盖率下降，自动补用例 |

## 创作方向建议

1. **「从提示词工程到Loop工程」**：梳理AI人机交互范式的三次跃迁——提示词工程（人写prompt）→ Harness工程（人搭框架）→ Loop工程（人设目标，AI自己循环）。分析Boris Cherny那句「不再自己写提示词，而是让Agent去提示Claude」背后的哲学。
2. **「四种Loop的设计哲学」**：深入拆解Anthropic公开的四种Loop原语，分析各自适用场景、工程复杂度、可靠性风险。可以结合Claude Code源码层面的设计讨论。
3. **「为什么软件工程先被打穿」**：代码场景天然适合Loop——目标可写成issue，过程可拆为文件修改，工具可跑测试，结果可用diff和CI验证。为什么其他场景（研究、表格、PPT、数据分析）也在跟进？
4. **「Loop不会自动带来可靠」**：一个设计很差的Loop只会让错误更快地自我复制。讨论Loop工程中的可靠性机制：上下文压缩、权限隔离、子Agent、hooks、skills、会话存储。
5. **「杀死提示词比赛」**：Loop做得越好，用户越不需要提示词技巧。普通用户不再需要把自己训练成提示词工程师——这会如何改变产品设计和用户-AI关系？
