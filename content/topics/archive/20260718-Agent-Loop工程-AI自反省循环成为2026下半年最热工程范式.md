# Agent Loop工程：AI自己干活、监工和返工

## 热度背景

2026年上半年，**Loop工程**（Agent循环工程）从一个工程实践逐步发酵为AI产业最热范式。2月Boris Cherny（Claude Code创始人）已在播客中讨论agent loop概念；5月在Sequoia AI Ascent公开宣称"loops are the future"；6月7日Google工程负责人Addy Osmani发表《Loop Engineering》长文正式命名这一范式；6月30日Anthropic官方博客《Getting Started with Loops》全面公开四种Loop原语；7月16日Boris发布《Steps of AI Adoption》五级成熟度模型，将Loop定位为通往AI-Native的关键台阶——这五个月，Loop工程完成了从少数人的实践到行业共识的转折。

与此同时，GitHub上83K+ Star的Ponytail插件（2026.6.12发布，DietrichGebert创建）、全球各Agent产品（Claude Code / Codex / ZCode / MiniMax Code / Workbuddy / Kimi Work）集体将「写—跑—看结果—再改」做成默认能力，信号非常明确：**Agent不再等你提示词，它在自己转了。**

### 核心概念

**Loop工程关心的是：如何让Agent围绕一个目标不断「循环」运作——理解任务→调取上下文→调用工具→观察结果→判断是否完成→没完成就继续改。**

Anthropic官方博客（2026.6.30）将Loop定义为"agents repeating cycles of work until a stop condition is met"，并按"你交出什么"分成四种：

| 类型 | 你交出什么 | 触发方式 | 停止条件 | 示例 |
|------|-----------|----------|----------|------|
| Turn-based（回合制） | 验证检查 | 用户prompt | Claude判断完成 | 常规对话，但用skills做自验证 |
| Goal-based（目标驱动） | 停止条件（/goal） | 手动prompt | 独立evaluator模型确认目标达成 OR 达到最大轮数 | 「把首页Lighthouse分数做到90以上，最多试5次」 |
| Time-based（定时触发） | 触发器（/loop, /schedule） | 固定时间间隔 | 你取消 OR 工作完成 | 每30分钟检查PR、修复CI失败 |
| Proactive（主动式） | 完整prompt（无人值守） | 事件或时间表，无人实时在场 | 每任务按goal停止，routine直到关闭 | 定时扫描bug报告→triage→修复→review，全自动 |

**核心机制**：Goal-based 的 `/goal` 命令使用一个独立的、更小的evaluator模型（默认Haiku）检查停止条件——等于把"做的人"和"判的人"分开，避免模型自己给自己打分。Proactive 则是上述全部原语的组合：`/schedule`（定时触发）→ `/goal`（定义完成标准）→ skills（验证标准）→ dynamic workflows（并行agent编排）→ auto mode（无需人工确认）。

## 风险与争议

Loop工程不是免费的午餐。其核心拥护者也反复警告：

1. **Token成本无上限**：Loop天然没有成本天花板——设计就是不停。Osmani警告"token costs compound faster than almost any developer expects"。Boris自己也承认"如果这听起来很贵，它就应该贵——Anthropic毕竟是卖token的"。
2. **无人值守 = 无人发现的错误**："a loop running unattended is also a loop making mistakes unattended"
3. **Comprehension Debt（理解债）**：系统提交了你从未读过的代码，理解差距越滚越大。Osmani最尖锐的警告："两个工程师运行同样的loop，一个更快地交付自己理解的工作，另一个根本不理解自己在交付什么"
4. **Evaluator模型的局限**：evaluator不能独立运行命令或读文件——它只能从对话输出判断。如果Claude从不打印测试结果，evaluator无法编造。arXiv论文（2607.00038）进一步警告model-as-judge的reward hacking风险
5. **Boris Cherny的7.16五级模型**提供了更务实的视角：从Gated（0）→ Assisted（~1个agent）→ Parallel（~10个）→ Supervised Autonomy（~100个）→ AI-Native（1000+个），每一步都需要新的guardrails，不能跳级

## 创作方向建议

**核心论点**：Loop工程的本质不是让Agent更智能，而是迫使人类把"怎么算做完"写清楚——这正在将软件工程师的角色从"写代码的人"变成"写stop condition的人"。

1. **「从写代码到写stop condition」**（主线）：Boris 8个月没手写代码、每天审几十个PR、Anthropic内部Claude之间Slack互聊协调——用递进叙事展示工程师角色的根本位移。核心张力：当你的主要工作从"实现功能"变成"定义什么叫做完"，工程能力的定义本身在变化。
2. **「四种Loop的解剖」**（技术深度）：深入拆解四种Loop原语，重点讲解`/goal`的evaluator模型分离机制（这是Loop的最关键技术洞察）、`/loop`与`/schedule`的session vs cloud区别、Proactive如何组合五种原语实现无人值守。
3. **「为什么软件工程先被打穿」**：代码场景天然适合Loop——目标可写成issue，过程可拆为文件修改，工具可跑测试，结果可用diff和CI验证——即可验证的stop condition天然存在。为什么其他场景也在跟进？因为它们正在被改造成"可验证的"。
4. **「Loop不会自动带来可靠」**：一个设计很差的Loop只会让错误更快地自我复制。讨论Loop工程的可靠性机制：maker/checker分离、上下文压缩、权限隔离、子Agent、hooks、skills、会话存储。引入Osmani的"comprehension debt"警告。
5. **「杀死提示词比赛」**：Loop做得越好，用户越不需要提示词技巧。普通用户不再需要把自己训练成提示词工程师——这会如何改变产品设计和用户-AI关系？

### 叙事结构建议

**递进案例线**：
- 第一层：Boris的日常（8个月不写代码、手机审PR）→ 展示"发生了什么"
- 第二层：Osmani的Loop工作流（自动化CI triage→子agent修→另一个agent review→人工只在merge时介入）→ 展示"怎么做到的"
- 第三层：Anthropic内部（Claude之间Slack互聊、100%代码由模型生成、全部SQL由模型写）→ 展示"做到极致会怎样"
- 第四层：回到核心问题——当loop帮你写了所有代码，你到底在做什么？答案是：**你在写stop condition。**
