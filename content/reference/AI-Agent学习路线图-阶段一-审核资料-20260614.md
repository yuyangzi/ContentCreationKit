## 一、Grill-Me 审核问答记录

### Q1：读者最终造出什么

**问题**：总纲已用天气 Agent 做钩子，阶段一 topic 只列了知识点列表，没有贯穿全文的工程目标。读者读完能造出什么？

**决策**：**C 方案——从零写一个 Mini Agent Loop**。不用任何框架，100 行 Python 实现完整的 Observe→Plan→Execute→Observe 循环。与总纲形成叙事弧：总纲让你意识到翻车是因为不理解底层 → 阶段一让你亲手造出底层。

---

### Q2：DeepSeek V4-Flash 的 FC 稳定性

**问题**：旧版 DeepSeek 文档有"Function Calling 不稳定"的警告。V4-Flash 是否受影响？

**验证结论**（经 librarian 查证）：
- 旧警告来自 DeepSeek V3（2024.12），已被 V3-0324（2025.03）修复
- V4-Flash 当前 FC 文档无任何稳定性警告
- `tool_choice="auto"` 模式稳定可用
- ⚠️ `tool_choice="required"` 在思考模式下返回 400（known issue）
- ⚠️ 多轮 FC 必须传回 `reasoning_content`
- `deepseek-chat` 别名将于 2026-07-24 废弃

**决策**：教程使用 `model="deepseek-v4-flash"`，显式禁用思考模式（`"thinking": {"type": "disabled"}`），用 `tool_choice="auto"`。避开 known issues。

**来源**：[DeepSeek API 文档](https://api-docs.deepseek.com/guides/function_calling/)，[GitHub Issue #1376](https://github.com/deepseek-ai/DeepSeek-V3/issues/1376)，[更新日志](https://api-docs.deepseek.com/updates)

---

### Q3：FC 新默认行为的处理

**问题**：2026 年 OpenAI/DeepSeek API 有多个新默认行为（`parallel_tool_calls` 默认开启、`strict: true` 结构化输出、`refusal` 字段、流式 tool calls），topic 描述的 FC 机制偏旧。

**决策**：**A 方案——渐进式**。先讲"理想情况"（一次一个 tool_call、无 refusal），让读者跑通 Mini Agent Loop。然后把 parallel/refusal/strict 作为"进阶坑"在 Harness Engineering 模块里一次性讲。符合"从坑出发"的定位。

### Q4：Mini Agent Loop 的工具选型

**问题**：总纲已用天气 Agent 做钩子，阶段一重复用天气会有重复感。选什么工具来展示 Agent Loop？

**决策**：**天气 + 文件读写**。理由：
- 文件读写是工程师最熟悉的操作，无需额外解释
- "Agent 能不能删我的文件"是读者瞬间能共情的焦虑 → 自然过渡到 Harness Engineering
- 渐进式叙事：先读文件 → 再加写文件 → 再问"要不要限制它能写哪些目录" → 引出 Harness

### Q5：模块一（LLM 基础）的深度

**问题**：Topic 要求"推理 vs 训练 → 讲透"，但转型工程师不需要理解推理管线（prefill/decode/KV cache）。真正的 Agent 不可靠性根因在"概率性生成 + Temperature 的影响"。

**决策**：模块一从"LLM 科普"收窄为"Agent 不可靠性根因分析"：

| 知识点 | 原要求 | 调整为 |
|--------|:--:|:--:|
| 推理 vs 训练 | 讲透 | 够用即可（知道"Agent 在推理模式下运行"） |

## 二、背景研究关键发现

### 2.1 Harness Engineering 概念验证

**结论：Harness Engineering 是一个定义清晰、来源丰富、可教授的成熟概念。**

| 来源 | 贡献 |
|------|------|
| OpenAI（2026.02） | 正式提出术语"Harness Engineering" |
| Martin Fowler & Birgitta Böckeler（2026.04） | 规范框架：Guides（feedforward）+ Sensors（feedback） |
| Anthropic（2026.03-04） | Generator-Evaluator 架构、Managed Agents、Sprint Contracts |
| Addy Osmani | Ratchet Principle：每次错误变成永久 harness 修复 |
| Mitchell Hashimoto | Agent = Model + Harness 公式 |

**核心框架**（Böckeler/Fowler）：
```
Guides（前置约束）：CLAUDE.md、System Prompt、Skills、Tool Definitions
Sensors（后置验证）：Linters、Tests、Hooks、UI Automation
```

**关键数据**：SWE-bench Pro — harness 改动带来 22+ 分的波动，模型升级仅约 1 分（同一模型在不同工具/架构下差距可达 42%→78%）。LangChain Terminal Bench 2.0：52.8% → 66.5%（仅靠 harness 优化，同样模型）。

**对教程的建议**：15% 模块聚焦"Minimum Viable Harness"——写好指令文件、加传感器、应用棘轮原则、设成本封套。不要深入学术论文（NLAH、SARC、ATLAS-RTC 等对 Stage 1 太深）。

**来源**：[OpenAI - Harness Engineering](https://openai.com/index/harness-engineering/)，[Martin Fowler](https://martinfowler.com/articles/harness-engineering.html)，[Anthropic - Harness Design](https://www.anthropic.com/engineering/harness-design-long-running-apps)，[Addy Osmani](https://addyosmani.com/blog/agent-harness-engineering)，[Mitchell Hashimoto](https://mitchellh.com/writing/my-ai-adoption-journey)

---

### 2.2 Function Calling API 技术准确性

**已验证的 topic 描述**：
- ✅ `tool_choice` 模式（auto/none/required/指定函数/allowed_tools）→ 正确，allowed_tools 为 2026 年新增
- ✅ `finish_reason` 值（stop/tool_calls/length/content_filter）→ 正确
- ✅ `tool_call_id` 绑定机制 → 正确
- ✅ Tool Schema 设计三原则（明确性/最小权限/幂等性）→ 正确

**Topic 遗漏的关键特性**：
- `parallel_tool_calls` 默认开启——模型可能同时返回多个 tool_call。⚠️ 与 `strict: true` 的兼容性取决于模型代际：gpt-4o 及更早不兼容，gpt-5+ 原生支持
- `strict: true`（Structured Outputs）——新的 schema 合规保证，旧 JSON mode 已是 legacy
- `refusal` 字段——模型可以合法拒绝调用
- `allowed_tools` 模式——第 5 种 `tool_choice` 选项，约束模型到工具子集
- Streaming tool calls——分块到达，需按 index 聚合
- Responses API（`/v1/responses`）——Chat Completions 的新替代

**建议**：教程先不讲这些（Q3 渐进式决策），但 topic 文件的技术细节部分应标注这些特性的存在。

**来源**：[OpenAI Function Calling 文档](https://developers.openai.com/api/docs/guides/function-calling)，[Structured Outputs 指南](https://developers.openai.com/api/docs/guides/structured-outputs)，[DeepSeek Function Calling 文档](https://api-docs.deepseek.com/guides/function_calling/)

---

### 2.3 最常遇到的 FC 陷阱（来自社区）

| 陷阱 | 影响 | 教程应覆盖 |
|------|------|:--:|
| `tool_call_id` 不匹配 | 结果被忽略 | ✅ 必须在代码中展示正确绑定 |
| 忘记附加助手消息 | 上下文断裂 | ✅ 展示完整的 messages 构造 |
| `content` 和 `tool_calls` 同时为 null | 拿到空响应 | ✅ refusal 检查 |
| 工具结果消耗 token 导致截断 | 回复不完整 | 🔶 提一下 |
| 嵌套深度 >3 层导致参数幻觉 | 调用失败 | — 阶段一不过度深入 |
