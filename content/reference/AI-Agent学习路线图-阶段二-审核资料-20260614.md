## 一、Grill-Me 审核问答记录

### Q1：项目结构（两个项目 vs 统一项目）

**问题**：Topic 有两个独立项目——模块一的"文档问答系统"和模块三的"三合一 Agent"。读者先搭一个，扔掉重来搭第二个。阶段一的成功经验是"一个项目贯穿全文"。

**决策**：**A 方案——统一项目**。全书只做一个"文档问答 Agent"。模块一搭 RAG 管线 → 模块二用 LangChain 重构 → 模块三给 Agent 加数据库查询 + 文件保存。一个项目从头长到尾。

---

### Q2：LangChain 模块深度

**问题**：Topic 计划逐一讲完 LangChain 六个模块，占 35%。但六个模块分 200 字每个，远远达不到阶段一的深度标准。阶段一用 47% 才讲透四件事。

**决策**：**收缩为 20%**。从"逐一讲解六大模块"变为"**从 Mini Loop 到 LangChain Agent**"。只讲三个有对比价值的模块：

| 模块 | 对比价值 | 处理 |
|------|----------|:--:|
| Agents（AgentExecutor） | 极高——直接对标阶段一的 Mini Agent Loop | ✅ 核心 |
| Memory | 高——阶段一的 Loop 没有持久记忆 | ✅ 核心 |
| Chains | 中——链式调用 vs while 循环 | ✅ 可讲 |
| Models | 低——换一个 API 调用方式 | 🔶 提一下 |
| Prompts | 低——阶段一的上下文工程已覆盖 | 🔶 提一下 |
| Indexes | 不需要——这是 RAG 模块的内容 | ❌ 移到模块一 |

---

### Q3：RAG 的教学方式（原生 vs LangChain）

**问题**：Topic 写的是"用 LangChain 搭建文档问答系统"。但阶段一的成功模式是"原生 → 框架"——先裸写，再用框架展示省了多少。

**决策**：**A 方案（原生+框架）**。模块一先裸写 RAG（ChromaDB 直接 API + sentence-transformers），再展示 LangChain 的 DocumentLoader + TextSplitter + VectorStoreRetriever 等价写法。形成和阶段一对称的叙事弧："还记得那 40 行代码吗？LangChain 10 行搞定。"

---

### Q4：数据源选择

**问题**：Topic 写的是"上传 PDF → Chunking"。但 PDF 解析（版面检测、表格提取）本身就是一个复杂话题，会分散 RAG 核心概念的教学。

**决策**：**txt/md 文本文件**。零解析复杂度，直接用 `open()` 读取。读者专注于 Chunking + Embedding + Retrieval 的核心流程。真实场景中读者可以自行替换数据源。

---

### Q5：最终 Agent 的工具集

**问题**：Topic 写的是"天气查询 + 数据库检索 + API 调用"。天气查询放在文档问答系统上很违和。

**决策**：**文档检索 + 数据库查询 + 文件保存**。三个工具对应三种数据层：

| 工具 | 数据层 | 衔接 |
|------|--------|------|
| 文档检索（RAG） | 非结构化数据 | 模块一的产出 |
| 数据库查询（SQLite） | 结构化数据 | 企业场景标配 |
| 文件保存 | 持久化 | 自然衔接阶段一的 `write_file` + Harness 权限控制 |

---

### Q6：模块组织

**问题**：RAG 的"原生→框架"弧是放在模块一内完成，还是跨模块一、二？

**决策**：**A 方案（RAG 自成一体）**。模块一内完成 RAG 的全弧（裸写 → LangChain 重构），读者在模块一结束时完成"我会 RAG 了"的里程碑。模块二讲 LangChain Agent+Memory，模块三实战集成。

---

### 内容修改

**修改 1（L22-26）**：模块布局

当前：
```
RAG 检索增强生成（40%）→ LangChain 六大模块（35%）→ 构建第一个单 Agent（25%）
```

建议改为：
```
RAG 裸写+LangChain重构（40%）→ LangChain Agent+Memory（20%）→ 多工具集成（32%）
                                                     ↑ 引子+结尾（8%）
```

**修改 2（L42）**：删除旧项目描述

当前：
```
**实操项目**：用 LangChain 搭建一个文档问答 Agent——上传 PDF → Chunking → 存入向量数据库 → 检索 → LLM 回答。
```

建议改为：
```
**统一项目**：全书只做一个"文档问答 Agent"，逐步生长。
- 模块一：裸写 RAG（ChromaDB 直接 API）→ LangChain 重构
- 模块二：用 LangChain AgentExecutor + Memory 包装
- 模块三：加数据库查询 + 文件保存
最终产物：一个能查文档、查数据库、保存结果的完整 Agent。
```

**修改 5（L46-62）**：LangChain 模块收缩

建议替换为：

```
### 模块二：从 Mini Loop 到 LangChain Agent（20%）

**定位：用 LangChain 重构你在阶段一写的 Agent Loop。**

| 知识点 | 核心内容 |
|--------|----------|
| Agent 创建 | `create_agent`（`langchain.agents`）——对比阶段一的 while 循环 |
| 对话记忆 | `checkpointer=InMemorySaver()`——直接传参给 `create_agent`，不需要额外包装 |
| 链式调用 | LCEL（`prompt \| model \| parser`）——对比阶段一的 messages 手动管理 |

**不讲的内容**：
- Models、Prompts（阶段一的上下文工程已覆盖基础，代码中出现时提一句即可）
- Indexes、VectorStores（已并入模块一 RAG 部分）

**关键认知**：LangChain 的 AgentExecutor 底层就是你在阶段一写的 while 循环。你花两天写的 100 行代码，LangChain 用一行 `agent.invoke()` 封装了。但你知道里面的 finish_reason 和 tool_choice 在干什么——这就是"原理先行"的价值。
```

**修改 6（L66-78）**：模块三工具集更新

建议替换为：

```
### 模块三：多工具集成——文档检索 + 数据库查询 + 文件保存（32%）

**定位：把 RAG Agent 升级为多工具 Agent。**

| 工具 | 实现 | 数据层 |
|------|------|--------|
| 文档检索 | 模块一的 RAG 产出 | 非结构化数据 |
| 数据库查询 | SQLite + 自然语言转 SQL | 结构化数据 |
| 文件保存 | 查询结果写入本地文件 | 持久化 |

**与阶段一的衔接**：
- 文件保存工具直接复用阶段一的 `write_file` + Harness 权限控制（ALLOWED_DIRS）
- 数据库查询展示 Agent 如何在两种数据层（文档 vs 数据表）之间切换
- 对话记忆由模块二的 Memory 统一管理
```

## 四、技术引用验证（2026-06-14，二次验证）

### LangChain API 当前状态 🚨

**验证版本**：`langchain==1.3.9`、`langgraph==1.2.5`、`langchain-core==1.4.7`、`langchain-openai==1.3.2`（2026-06-12）

| 旧 API | 状态 | 当前替代 |
|--------|:--:|------|
| `create_react_agent`（langgraph.prebuilt） | 已废弃 | `from langchain.agents import create_agent` |
| `RunnableWithMessageHistory` | 已废弃 | `checkpointer=InMemorySaver()` 直接传参 |
| `AgentExecutor` | 已废弃（仍可运行） | `create_agent()` |
| `ConversationBufferMemory` | 已移除 | checkpointer（无需额外 memory 层） |
| `LLMChain` | 已废弃 | LCEL：`prompt \| model \| StrOutputParser()` |

**Memory 正确做法**：
```python
from langchain.agents import create_agent
from langgraph.checkpoint.memory import InMemorySaver
agent = create_agent(model=..., tools=[...], checkpointer=InMemorySaver())  # 教程用；生产请换 PostgresSaver
agent.invoke({"messages": [...]}, {"configurable": {"thread_id": "..."}})
```

**API 签名关键变化**：

| 变化项 | 旧 API | 新 API |
|--------|--------|--------|
| 系统提示 | `prompt=` | `system_prompt=`（`str \| SystemMessage`，教程用 string 即可） |
| 工具传参 | `tools=ToolNode([...])` | `tools=[t1, t2]` |
| 状态 Schema | `state_schema=BaseModel` | `state_schema=TypedDict` only |
| 运行时上下文 | `config["configurable"]["key"]` | `context=Context(...)` + `context_schema=Context` |
| 流式节点名 | `"agent"` | `"model"` |

**⚠️ 流式注意事项**：
- `stream_mode="messages"` 存在间歇性重复文本 bug（Forum 2025.11），教程推荐用 `stream_mode="updates"`
- 必须加 `version="v2"` 获得统一输出格式（`stream()` 场景下需显式指定；`astream_events()` 已默认 v2，LangGraph ≥ 1.1）
- **Python 3.10/3.11 下 `create_agent` 异步流式不工作**（只返回完整消息不分块），需 **Python ≥ 3.12**

来源：[LangChain 官方文档](https://docs.langchain.com/oss/python/migrate/langgraph-v1)、[LangChain Forum](https://forum.langchain.com/t/2284)
