# Loop 范式，人类对 AI 入侵的再一次退后

> 六月第一周，Claude Code 创建者说自己不写 prompt 了，OpenAI 的人在推 loop，Google 的人给这股实践起了名字。而真正的起点，是一个澳大利亚开发者在 2025 年 7 月写的一行 bash——`while true; do claude -p "$(cat prompt.md)"; done`。从 prompt 到 loop，人类又一次往抽象层级的楼上爬了一层。

2026 年 6 月第一周，三件事在 5 天内接连发生。Claude Code 创建者 Boris Cherny 在 Acquired 播客上说他已经卸载 IDE、不写 prompt、只写 loop。两天后，OpenAI 的 Peter Steinberger 在 X 上宣告"你该设计 loop 而不是写 prompt"。又过了一天，Google 工程主管 Addy Osmani 给这股实践起了名字——Loop Engineering。

不是商量好的。是行业共识在同一时刻的收敛。

6 月 25 日，36氪报道了黄仁勋的发言——"Prompt 正在过时，Loop 才是新范式"，中文科技媒体炸了锅。但把 Loop 看作 6 月的新闻就错了。

往回推 11 个月。

2025 年 7 月，一个叫 Geoffrey Huntley 的澳大利亚开发者写了一行 bash：`while true; do claude -p "$(cat prompt.md)"; done`。他管它叫 Ralph Wiggum Loop，名字来自辛普森一家里那个看着傻乎乎但凭坚持把事情做了的角色。核心原则简单到粗暴：每次迭代全新上下文，状态只活在文件系统里，一次只做一件事，`--yolo` 模式不给权限确认。

当时大多数人觉得这是一个疯子的玩具。

2025 年 11 月，Anthropic 把 Ralph 做成了 Claude Code 官方插件。2026 年春天，`/goal` 命令在 Claude Code 和 Codex 中同时出现——"跑到条件满足为止"，独立的小模型做验收。写代码的模型不能给自己打分，验收和被验收必须分开。然后才是 6 月的井喷。Boris 说他管理"几百到上万"个 agent 同时工作，8 个月没手写一行代码——他曾经一个月内提交了 259 个 PR，没一个是他自己写的。SemiAnalysis 的报告显示 4% 的 GitHub 公开提交来自 Claude Code，预计年底超 20%。

从 `while true` bash 一行到 Loop Engineering 六块积木，整整 11 个月。

---

回到一周前留下的一个问题。

一周前我写了《AI 时代：压缩了执行力，放大了判断力》。核心结论来自 Anthropic 的数据：AI 把"做出来"的执行力差异压平了，所有人的手速被拉到同一水平线，溢价全在"判断对错"上。**人决定造什么，AI 决定怎么造。**但结尾留了一个追问：万一 AI 把判断力也学会了呢？

两天后写了《AI 的决策半径正在变大》。GPT-5.4 进实验室自己找催化剂，医疗模型 MIRA/AMIE 进病历系统做诊断决策超越医生，一个 80 年的数学猜想被推翻。结论变了：**AI 不只是在执行，它开始做判断了。**人的角色从"拍板者"变成了"定义拍板标准的人"。

如果在这里停下来，会有一个挥之不去的矛盾。第一篇说判断力是护城河，第二篇说 AI 正在翻护城河。那人的护城河到底还在不在？

Loop 就是这个矛盾的答案。

---

判断力没有被取代。判断力被**工程化**了。

在没有 Loop 之前，你用 AI 写代码的流程是这样的：写一个 prompt → 等 Claude 出结果 → 读代码 → 判断对不对 → 再写一个 prompt → 再读 → 再判断。每一步都是你在推。代码是 AI 写的，循环本身是你在跑。你是一台"判断引擎"——每轮迭代都要重新启动，每轮的上下文都要你重新喂。

**在 Loop 出现之前，人本身就是循环。**

Loop Engineering 做的事，不是让 AI 变聪明，而是把"人这个循环"工程化成一个能自己跑的系统。你不再是循环的执行者，你变成了循环的设计者。你把"什么算完成""什么算跑偏""什么时候停"写成规则，然后系统自己转。

Osmani 在 6 月 7 日的博客里用一张对比表说明了区别。传统 Prompt Engineering 的流程：人手写 prompt → 读结果 → 下一轮，每一环都是人的手在推。Loop Engineering 的流程：Automation 触发 → Sub-agent 执行 → 另一个 Sub-agent 验证 → State 记录 → 下一轮。人只设计了规则，不参与每次运转。五块积木加上 State 作为外部记忆：

1. **Automations** — 做发现和分流
2. **Worktrees** — 做并行隔离
3. **Skills** — 固化工程知识
4. **Connectors** — 接外部工具
5. **Sub-agents** — 把写代码和验收拆开

每一块都是为了替代原本人手动做的某个环节。

这不是 AI 变强了多少的问题。是人的角色又一次往上跳了一个抽象层级。

---

如果觉得这个概念太抽象，看演化线。

2022 年姚顺雨团队发表了 ReAct 论文，首次提出推理和行动可以交织成一个循环。这是学术地基。2023 年 AutoGPT 横空出世，自主 agent 第一次被大众看到，但混乱到没法用。

真正的转折在 2025 年 7 月。Huntley 那行 bash 解决了一个结构性问题：怎么让 AI 持续跑，又不让上下文膨胀到崩溃？答案出奇地简单：每次迭代清空上下文，状态只存在文件系统里。这和今天的 Loop Engineering 在原理上一脉相承——不靠 AI 自己"记住"，靠外部 state 来管理。

然后 2026 年春天，`/goal` 命令出现了。跟 `/loop` 按时间间隔重复不同，`/goal` 跑的是条件驱动——持续执行直到目标满足，每轮由独立小模型判断"完成了吗"。写代码的模型不能自己判自己的卷子，这是整个设计里最关键的一笔。

然后是 6 月的井喷。Boris 在 Acquired 上说"我不 prompt Claude 了，我的 job 是写 loop"。接着是 Steinberger 的推文，Osmani 的命名，Jensen 的宣言。Codex 和 Claude Code 两套独立开发的系统，loop 原语几乎一模一样——自动化、worktree 隔离、sub-agent 分离、外部 state 管理。

**行业共识是长出来的，不是商量出来的。**

---

退一步看。

Loop Engineering 这个命名其实不太准确。叫 Loop 容易让人觉得是一种新的编程范式，但它的本质不在代码里，在角色转换上。

每一次范式迭代，本质都是在重新划分人和机器的边界。

写代码的时代，人把每一步逻辑都写清楚。到了 prompt 时代，人描述目标，AI 补全步骤。到了 context 时代，人不用每次重说一遍——工程知识被固化在 AGENTS.md 和 SKILL.md 里。到了 harness 时代，人设计了自动化触发和调度体系，你已经部分从循环里退出来了，只是还在按按钮。

Loop 是这一步的彻底完成。你连按钮都不按了。你设计规则，系统自己跑。

但一个细节容易被忽略。Loop 替代的不是人的"判断"，是人的"判断动作"。你把"每次读代码然后决定要不要改"这个机械动作交出去了，但"什么算好代码""什么算完成"的标准还是你定的。`/goal` 命令需要一个验收模型来判定"跑完了没"——验收标准的定义者还是人。

这就是前两篇矛盾的解法。第一篇说判断力是护城河，第二篇说 AI 也在做判断——两者不矛盾，因为 AI 做的判断和你做的判断不在同一个抽象层级。AI 做的是"这一行代码对不对"，你定的是"对的标准是什么"。Loop 把前者吞了，但碰到了后者的玻璃天花板。

---

问题正在这里。

如果你把判断标准写成 rule、写成验收脚本、写成 SKILL.md——这些标准本身，离被自动化还有多远？

Boris 说他的工作是写 loop。但写 loop 这件事，本质上也是一种工程行为——定义触发条件、验收标准、失败回退路径。如果有一天 AI 能观察你写的 loop 运行质量，自动调整验收阈值，自动优化 agent 团队的配比和并行策略——那你连"设计 loop"这件事也可以被 AI 做了。

那时候，人还能设计什么？

Osmani 在文章末尾写了一句很重的话："Build the loop. But build it like someone who intends to stay the engineer, not just the person who presses go."

坦白讲，这句话让我想了很久。Loop Engineering 最危险的地方不在于它会让工程师变懒。在于它太好用了，好用到你很容易忘记自己才是设计者。一旦你停止思考设计本身，你就不再是工程师了。你是按钮操作员。

这也是前两篇一直在追问的问题的延续。执行力被压缩之后，人剩判断力。决策半径扩大之后，判断力也开始被 AI 渗透。Loop 把判断力工程化——但如果你连工程化判断力的标准都懒得维护了，你还能剩什么。

---

Huntley 用辛普森一家的 Ralph Wiggum 给循环命名——一个笨拙但凭坚持把事情做了的角色。这个隐喻指向一个有趣的张力：Ralph 是不理解但执行，而 Loop 的逻辑正好反过来——是你理解之后把标准写下来，让它替你照着做。两者之间隔着一道线：一边是"不思考所以机械"，一边是"思考完毕后把机械动作外包"。

这个区分太细了，细到容易被淹没在"AI 取代人类"的大叙事里。但它恰好是整个故事里最关键的一条线。

从代码到 prompt 到 context 到 harness 到 loop，每一次"过时"都不是技术的胜利——是人的角色往抽象层级又爬了一层。你不会写代码了→但你学会写 prompt 了→你不会写 prompt 了→但你学会设计 context 了→你不会管理单个 agent 了→但你学会设计 loop 了。

模式总在重复：每放弃一个旧身份，就捡起一个新身份。

问题是，这次爬到的地方离天花板还有多远。上次爬到 prompt 的时候，你觉得判断力是护城河。这次爬到 loop 的时候，你发现判断力的执行部分也可以外包了。下一次再爬一层的时候——如果连设计判断标准都变成了 AI 可以在另一个 meta-loop 里自己迭代的东西——

人还能爬到哪里去。
