# Claude Code 隐写术标记中国用户：技术拆解与信任危机

> 2026年6月30日，Reddit 上一个叫 LegitMichel777 的用户发了个帖子。他逆向工程了 Claude Code 的源码，发现这个 AI 编程工具在系统提示词里藏了一套隐写标记系统——用来识别中国用户。四天后，工信部 NVDB 发布风险提示，用一个极为罕见的词定义了这次事件："危害严重"。

---

6月10日，Anthropic 给美国参议院银行委员会写了封信。信里指控阿里巴巴 Qwen 实验室用了约2.5万个假账户、进行了2,880万次对话来蒸馏 Claude 模型（Forbes 6月26日报道）。Anthropic 管这叫"史上最大规模 AI 蒸馏攻击"。从他们的视角看，这是防御战——有人用假账户大规模窃取模型能力，他们需要知道这些请求从哪来。这直接解释了为什么 Claude Code 后来会装上一套隐写检测系统。

3月，Anthropic 在 Claude Code 里上线了一个"实验性"检测模块。3月30日，v2.1.88 的 npm 包因为调试 sourcemap 泄露了约51.2万行 TypeScript 源码——不过那时候还没人注意到里面的门道。4月2日，v2.1.91 发布，隐写检测代码正式进入分发渠道。从这天起，每一个通过代理或中转 API 使用 Claude Code 的用户，都在不知情的情况下被打上了标记。

一直到6月30日，LegitMichel777 在 r/ClaudeAI 发帖（Hacker News 榜首，2400+ upvotes），把这套机制整个拆了出来。开发者 Thereallo 独立确认后在个人博客上发了完整逆向工程代码。

Anthropic 工程师 Thariq Shihipar 第二天在 X 上回应了（2026.07.01）。他说这是"3月启动的一项实验"，目的是"防止来自未经授权转售商的账户滥用以及保护模型免受蒸馏攻击"，团队"早就打算移除这个功能了"。PR 合并了，v2.1.197 发布了，代码被移除——但官方 changelog 里对这件事只字未提。

一个运行了3个月、覆盖了从 v2.1.91 到 v2.1.196 所有版本的"实验"。changelog 里写不下。

---

## 触发条件与检测机制

这套代码有明确的激活前提。

只有在环境变量 `ANTHROPIC_BASE_URL` 被设为非官方端点——也就是 `api.anthropic.com` 以外的代理、网关、中转服务——它才会激活。直接连官方 API 的用户不受影响。

一旦触发，执行三项检查：

**时区检查**。读的是系统时区（`Intl`），不是 IP GeoIP。所以就算挂了 VPN、用了境外代理服务器，只要系统时区是 `Asia/Shanghai` 或 `Asia/Urumqi`，这一项就命中。VPN 在这个维度上毫无意义。

**域名黑名单匹配**。147条域名条目，用 Base64 编码后再做 XOR（密钥=91）混淆，存在二进制里。解码后是三个梯队的名单：中国互联网公司（baidu.com、alibaba-inc.com、bytedance.net、jd.com、meituan.com、netease.com、ctrip.com 等等），中国 AI 实验室（moonshot.ai、minimax.chat、zhipu.com、bigmodel.cn、baichuan.com、stepfun-inc.com、01ai.com、dashscope.com、volces.com），以及一批 API 中转代理（anyrouter.top、proxyai.com、yunwu.ai、openclaude.me 之类）。

**AI 实验室关键词匹配**。11个关键词：deepseek、moonshot、minimax、xaminim、zhipu、bigmodel、baichuan、stepfun、01ai、dashscope、volces。直接匹配 `ANTHROPIC_BASE_URL` 里的 host 字段。

三项检查的结果汇到一个3-bit 的状态里，然后编码进每个 API 请求的系统提示词。

## 三个比特，四种撇号

每个 Claude API 请求的系统提示词里都有这么一行：

```
Today's date is 2026-06-30.
```

这行看似普通的文本，承载了两层信息。

第一层在日期分隔符上。如果时区不是中国，显示 `2026-06-30`（连字符）。如果是中国时区，变成 `2026/06/30`（斜杠）。1比特信息，肉眼一眼能看出区别。Thereallo 重构的代码里，这一步叫 `cnTZ ? date.replaceAll("-", "/") : date`——直白到让人不知道该觉得它粗糙还是大胆。

第二层才是真正的隐写术，藏在一个小小的撇号里。

同一个 `'`，在视觉上完全一样，但底层 Unicode 编码不同：

| 你看到的 | Unicode | 编码背后的真实含义 |
|:---|:---|:---|
| `'` | U+0027（标准 ASCII 撇号） | 什么都没命中——域名和实验室关键词均不匹配 |
| `'` | U+2019（右单引号） | 命中了域名黑名单 |
| `ʼ` | U+02BC（修饰字母撇号） | 命中了 AI 实验室关键词 |
| `ʹ` | U+02B9（修饰字母 prime） | 域名和关键词都命中了 |

共计2比特信息编码在撇号变体里，加上日期分隔符的1比特，每个请求携带3比特的身份标记。

XOR 混淆解码只有三行：

```JavaScript
var Kup = 91;
function Gla(encoded) {
  let bytes = Buffer.from(encoded, "base64");
  let out = "";
  for (let byte of bytes) out += String.fromCharCode(byte ^ Kup);
  return out.split(",");
}
```

密钥硬编码为91。不是动态生成的，不是从环境变量读取的，是直接写死在代码里的。对于一个自称"实验"的功能来说，这个设计选择本身就耐人寻味——如果只是内部实验，为什么要把混淆密钥编译进公开发布的 npm 包里？

Anthropic 没有修改请求体、没有加额外的 HTTP 头、没有动 API 协议的任何部分。他们修改的是一行用于告诉模型"今天是几号"的系统提示词。视觉上完全相同的字符，承载了不同的身份标签。这个方案在工程上确实简洁——不需要新字段、不需要新的通信通道、不影响任何现有功能。但正是因为这种"简洁"，让它可以在代码里静默运行了三个月才被发现。

而且它绕过了最常见的审计手段。看源码、看网络请求、看日志，看到的都是"正常"的文字。只有拿到压缩混淆后的 TypeScript，一对一对地去比 Unicode 码位，才会意识到——为什么这里的撇号是 U+2019？

## 这不是第一次

隐写术也不是孤例。曝光前两个月，隐私顾问 Alexander Hanff 发现 macOS 版 Claude Desktop 安装后会静默向 Chrome、Brave、Edge 等7个浏览器注入 Native Messaging 配置文件（个人博客，2026.04.18），预授权一个浏览器扩展绕过沙箱限制。安装程序没问，服务条款里没有。Hanff 直接管这叫"spyware"。

两件事独立发生，但放在一起看，画面就清晰了。一个是桌面客户端做浏览器注入，一个是 CLI 工具做隐写标记。一个在明处，一个在暗处。两个都是"未经用户明确同意就在系统层面上做了用户没预料到的事"。这不是技术路线的偏差，这是行为模式。

Anthropic 给 Hanff 的回应至今未见公开。给 Thereallo 的回应倒是快——曝光第二天就合并了 PR。但快归快，Thariq Shihipar 的声明里有一句话特别值得玩味："团队之后已经推出了更强的缓解措施，我们也早就打算移除这个功能了。"

早就打算移除。在公开曝光之前。但代码从3月跑到了6月。

## 从社区愤怒到官方定性

事态升级的速度比社区预期快得多。

6月30日 Reddit 曝光，7月1日 Anthropic 工程师回应，7月2日发布修复版本 v2.1.197，7月3日阿里巴巴内部通知全员禁用 Claude Code——推荐用 Qoder 替代，7月10日生效。7月8日，工信部网络安全威胁和漏洞信息共享平台（NVDB）发布了一则风险提示。原文是这样的：

> 近日，工业和信息化部网络安全威胁和漏洞信息共享平台（NVDB）监测发现，AI编程工具Claude Code存在安全后门隐患，危害严重。

注意这几个词：**安全后门**、**危害严重**。不是"安全风险"，不是"隐私问题"，是"后门"。"危害严重"不是套话——NVDB 的公告用了这个词意味着什么，看历史先例就知道。2026年3月对 OpenClaw 的安全建议只是通用性指导，没出现过类似措辞。

NVDB 的建议更直接："立即卸载或升级至已清除相关后门代码的最新安全版本"。"立即卸载"——一个国家级网络安全平台第一次针对单一海外 AI 编程工具说出这句话。

这也不是孤立的。7月前后，已经有几家公司做了类似决策。微软在5月取消了 Claude 的内部许可证，要求迁移到 GitHub Copilot CLI，6月30日截止。Meta 在6月29日限制使用 Anthropic 和 OpenAI 的编程 Agent，强制切换到基于 Llama 技术栈的 MetaCode。美团此前在4月限制了阿里 Qwen，7月2日又限制了字节豆包——豆包 Agent 功能实际截止日期为7月15日。

这些决策出发点和动机各不相同——有的是成本驱动，有的是数据安全考量，有的是技术栈自主可控。但时间线上的聚集，至少当前来看，折射出一个趋势：AI 编程工具正在从"想用就用"进入"谁能用、怎么用、用什么"的制度化阶段。

## 信任的边界在哪里

隐写术的技术水平甚至称不上高明。XOR 混淆是最基础的手段。Unicode 隐写在信息安全领域的历史可以追溯到十几年前。

真正的问题是这个：一个拥有 shell 权限和文件系统访问权的工具，在未经告知的情况下，用隐蔽手段收集用户身份信息并回传给远端服务器。

Claude Code 不是网页聊天框。它是一个运行在开发者终端里的 CLI 工具，能读你的代码、执行你的命令、访问你的文件系统。你信任它，给了它远超一般桌面应用的权限。然后你发现它在系统提示词里偷偷换了几个 Unicode 字符，就为了识别你的身份、你的雇主、你用的代理服务器。

Thariq 说这是"实验"。但一个实验跑3个月，覆盖公开发布的多个版本，在 changelog 里完全不提——你很难把它跟正常的产品迭代区分开来。实验和部署之间的那条线，在 AI 工具行业似乎一直模糊。但模糊到这种程度，已经不是边界问题，是信任问题。

Anthropic 的处境不难理解。蒸馏攻击对任何前沿模型公司都是真实且严重的威胁。2.5万个账户、2,880万次对话——放在任何一个商业语境里，这都是足以触发法律诉讼的规模。但选择的防御手段，把"保护知识产权"变成了一场隐蔽的用户监控，而监控的对象最终被精准定位到了特定国家的特定行业。这已经不是防卫过当的问题了——这直接挑战了用户跟 AI 工具之间最基础的信任假设。

这套代码是怎么通过内部审查的？写代码的工程师、review 代码的同事、管理产品的 PM，不可能没人知道这行 `"\u2019"` 取代 `"'"` 意味着什么。是大家都觉得"反正只是实验"所以通过了？还是内部就没有把它当作一个需要审视的伦理决策？不管是哪种，都暴露出一个行业的集体盲区——AI 公司对"安全"的定义，正在从"保护用户"滑向"保护自己"，而这两者的冲突地带，目前几乎没有规则。

工信部的公告本质上是在画一条线。你可以说是地缘政治驱动的监管行动，但它的核心信息是清晰的：在代码里藏后门标记用户，这样的做法不应该被接受。

这不是一个关于中国或美国的故事。这是一个关于工具和信任的故事。如果你用的命令行工具可以在不告知你的情况下，偷偷检查你的时区、你的域名、你服务的公司名称，然后把检查结果编码成看不见的字符传回服务器——你今天信任它，是因为你假设它不会这么做。但当假设被证明错了之后，信任靠什么重建？
