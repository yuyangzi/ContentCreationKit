# Agent 规模化 = 分布式系统 × LLM 不确定性

Agent 的热闹是公开叙事的热闹，Agent 的冷是生产环境的冷。88% 的试点从未进入规模化部署。不是因为 AI 不够强，是因为 Agent 规模化卡住的原因，恰好是软件工程过去二十年以为自己已经解决了的问题。

2026 年 7 月初，InfoQ 接连发了两篇相关文章。一篇《Agent 狂欢下的冷思考》（7 月 2 日），一篇《Agent 上岗之后，企业如何治理硅基团队？》（7 月 3 日）。

放在同一周，Claude Code 的独立年化收入被 FutureSearch 估算到了 $8B（其他机构估算在 $2.5B–$8B 之间），百万级月活 session，Anthropic 自己发了篇论文分析约 40 万个交互会话。AI 编程工具从"开发者的玩具"变成了"企业基础设施级收入"。

叙事在加速，这一点没什么争议。

但说了一个不同的数据。IDC 与联想的调研显示，88% 的 AI 试点从未进入规模化部署——每 33 个 POC 里只有 4 个真正跑在生产环境。Gartner 预测 40% 以上的 agentic AI 项目将在 2027 年前被取消，不是"可能"，是预测直接被取消。

KPMG 追踪了 237 位美国高管，发现 Agent 部署率从 2024 年初的 11% 升到了 2026 年 Q1 的 54%，但这里面有多少是真跑在关键链路、有多少是挂着 Agent 的名字跑自动化脚本——业界对"部署"的口径差异大到离谱，S&P Global 的同类统计只有 31%。

一个更扎眼的对比来自 GitLab。他们 2026 年那份覆盖 1500 人的调研给了这样一组数字：78% 的人说编码变快了，85% 的人说瓶颈从编码转移到了审查和验证，79% 的人说整体交付并没有加速。

编码快了，交付没快。瓶颈从"写代码"移到了"搞清楚这代码到底干了什么"。

## 编码快了，交付没快

GitLab 这组数字不是孤立信号。我在上一篇文章(AI速度鸿沟-写代码快了交付没提速)里详细拆解过这条管道的衰减曲线——CEPR 追踪了 10 万+ 开发者的数据，AI 让 commit 产出涨了 180%，真正发布出去的版本只多了 30%，终端用户端的使用量没有显著变化。180% 到 50% 到 30% 到零。审查、测试、构建、部署、排障——每一环都在漏。

这里不重复那篇文章的论证，只补充一个 Agent 规模化场景里特有的维度。

单人开发场景里，AI 编码快了对个人体验整体是正向的——你自己写的代码也得 review，AI 生成的多 review 一遍也没差太多。

但 50 个人的团队各自用 Claude Code 做事，结果是什么？InfoQ 那篇治理文章里提了一个关键观察：AI 天然具有"反协作"特征。人和 AI 的对话是个体化的，每个人和 Claude Code 的交互方式、生成的代码风格、对同一个问题的理解角度都不一样。50 个人 × 50 种 AI 使用方式 = 不是 50 倍产出，是 50 种需要重新对齐的代码。

吴恩达在 AI Fund 的实践印证了这一点：代码原型开发速度快了 10 到 100 倍，但产品管理、法务、合规成了新瓶颈。"一个功能一天写出来，但要等一周法务签字。"

这些瓶颈在 Agent 规模化场景里会被进一步放大——因为 Agent 的输出不只是"快但需要 review"，而是"快且不确定"。你面对的不仅是审查带宽不够，而是连"该审什么"都变得更难判断。

## 四道墙

把 Agent 规模化落地的问题拆开看，有四个维度反复出现。InfoQ 两篇文章分别从技术和治理角度覆盖了这些点，行业里的数据也在持续往同一个方向指。老实说，看完这些数据之后，我最大的感受不是"问题好严重"，而是"这些问题其实一点都不新"。

### 数据墙

OutSystems 2026 年的报告给了两个数字放在一起看：96% 的组织已经在某种程度上使用 AI Agent，但仅 12% 已经建立集中式平台来管理 Agent 蔓延。

不是 96% 和 70% 的落差，是 96% 和 12%。这意味着绝大多数组织是在缺乏统一治理框架的前提下让 Agent 去理解数据、做决策、执行任务。Agent 的能力再强，读的数据是脏的、碎片化的、跨系统不一致的，生成的结果就很难保证可靠。这当然不是 Agent 的问题——但从用户视角来看，Agent 输出的结果不对，锅背在"AI 不好用"上。

数据治理、主数据管理、跨系统的一致性同步——这些是典型的"砸钱砸人砸时间"项目，和 AI 叙事里的"一周上线"不是一个节奏。

### 监控墙

分布式系统的可观测性有一套成熟的工具链：Prometheus 管指标，Jaeger 管链路追踪，Grafana 管可视化。排查一个微服务的问题，你能看到哪个服务调了哪个服务、耗时多少、返回了什么、哪里报了错。

Agent 的场景完全不一样。Agent 做的事情是：接收输入，内部推理，调用工具，再推理，再调用工具，输出结果。传统的可观测工具只能看到工具调用的部分——调了哪个 API、返回了什么数据。但 Agent 为什么在这一步调这个工具、为什么下一步选了另一个方向、中间的推理链是什么样的——这些是黑箱。

Forrester 的数据显示，仅 38% 的生产环境 Agent 有自动化评估机制。超过六成的 Agent 在生产环境跑着，但没有人能系统性地判断它做得对不对。

Agent 需要专门的运行时基座，就像软件需要操作系统、云原生需要 K8s。传统工具（K8s、Serverless Container、Lambda）在面对 Agent 时都"拧巴"，因为它们是为确定性任务设计的。一个 HTTP 请求进来，处理完返回，生命周期结束。

Agent 的任务生命周期完全不是这个逻辑——它可能跑 30 秒，也可能跑 5 分钟；可能在中间的某一步等一个外部系统的结果，也可能在推理循环里卡住；可能在某一步产生了一个"看起来对但实际错"的中间状态然后继续往下走。

你没法用监控 HTTP 延迟的工具去监控一段推理链。这一点让我印象很深——整个现代运维体系都建立在"系统是可复现的"这个假设上，而 Agent 恰恰把这个假设砸了。

### 成本墙

用一个简单的算式就能说清楚。

假设一个 Agent 任务有 10 步，每一步的可靠性是 95%。端到端的成功率是多少？不是 95%，是 0.95 的 10 次方，大约 60%。如果是 8 步，每步 85% 可靠——更接近 Anthropic 论文里中级用户的实际水平——端到端成功率只有 27%。

这意味着一个看起来"AI 跑通了"的流程，实际跑十次只有六次真的全部跑通。另外四次要么中途失败要人工介入，要么跑完了但结果不对需要回滚，要么跑出来的东西看起来对但过了两天发现埋了个 bug。

InfoQ 治理文章里给了一个关键的公式：Token 真实成本 = Token 单价 / 任务成功率。你付的不只是每一千 token 的价格，你还得为那些失败的任务买单。而且是除法关系，不是乘法。

Anthropic 的研究还发现了一个有意思的变化趋势：Claude Code 用户的时间分配在 7 个月内发生了明显改变，Debug 的占比从 33% 降到了 19%，更多时间花在了运营和数据分析上。表面看是好的，但仔细一想——Debug 占比下降可能是因为用户学会了避免复杂任务，或者降低了验收标准。毕竟新手成功率只有 15%，你不能指望一个月的使用就把可靠性从 60% 拉到 90%。

### 信任墙

AGAT 2026 年的数据：82% 的高管信任自己组织的 Agent 安全策略，但仅 14% 的生产环境 Agent 通过了完整的安全审批流程。

82% 说"我相信"，14% 说"我核验过"。

这个落差不是高管"说一套做一套"。更可能的原因是：安全审批流程本身跟不上 Agent 的迭代速度。一个传统软件功能上线前，安全评审可以基于确定的功能边界做风险评估。Agent 的行为边界不确定——同一个 prompt 跑两次可能走两条不同的工具调用路径。你怎么审批一个边界不确定的东西？

GitLab 调研里还有一个数字：83% 的受访者将 AI 生成代码的积累视为安全隐患。绝大多数开发者不信任 AI 生成的代码，但他们还是在用——因为"快"的诱惑太大了。这种"边用边怕"的状态，本身就说明信任机制没有建立起来。

## 不是加法，是乘法

Agent 规模化落地的每一道墙，不是分布式系统难题再加一个 LLM 不确定性。是前者的难度被后者的不可预测性放大了。

数据墙不是"数据治理本来就难"加"AI 不理解数据"。数据本身就不干净，而 Agent 以一种不可预测的方式去理解这些不干净的数据——两个偏差叠加。传统的 ETL 出错至少报个异常，Agent 可能在脏数据上推理出了"正确的错误结论"，然后平静地交给了下一步流程。

监控墙不是"可观测性本来就复杂"加"推理链不透明"。是你用为确定性系统设计的追踪工具去抓一个非确定性过程的快照。拍到的是那一刻的状态，但 Agent 可能在下一轮推理中选择了完全不同的路径。追踪的意义在于复现，但当系统本身不可复现，追踪就变成了一堆没有行动指引的日志。

成本墙不是"算力调度要优化"加"Token 消耗要控制"。算力调度至少可预测，你知道一个 Pod 大概消耗多少资源。Token 消耗不可预测——同一个任务因为推理路径不同，消耗量可能差一个数量级。预算模型的前提是输入可预测，Agent 把这个前提干掉了。

信任墙是这四道墙里最难办的。分布式系统的故障模式虽然复杂，但至少可以枚举——网络分区、节点宕机、磁盘满、内存溢出——每一种都有对应的处理策略。Agent 的故障模式无法枚举，你没法穷举一段不可预测的推理链上所有可能出错的点。

K8s 管得了 Pod，管不了推理链。你能声明"我想要三个副本"，但你不能声明"这条推理链的风险等级不得高于 X"。

华为在 ACE Harness 平台上做了一件事值得注意：他们在 Agent 治理体系里引入了"蓝军 Agent"——专门跑对抗性任务、反向审视其他 Agent 输出的机制。这个思路说明了一件事：管理确定性的系统靠规则，管理不确定性的系统需要对抗。你没法用一个"如果 A 则 B"的规则去约束一个不可预测的推理过程，但你可以让另一个 Agent 专门盯着它挑刺。

但这个方案也揭示了问题的边界——即使是华为这种体量的公司，也只能做到"增加一层对抗性验证"，而不是"从根本上解决不确定性的治理问题"。

---

行业的喧嚣和数据的沉默之间，隔着的不是"技术还不够好"。是一个更根本的东西——我们用来承接 Agent 的基础设施，从监控、调度、安全审批到成本核算，全部建立在"系统行为可预测"这个前提之上。Agent 把这个前提打碎了。

那些在 Agent 规模化上走得相对靠前的团队，做法有一些共性：命名 ownership、限定 scope、自动化评估、敢发敢回滚。但这些是局部解。它们能帮一个团队活得比另一个团队好一点，但解决不了"分布式系统 × LLM 不确定性"这个乘数效应本身带来的结构性问题。

有一件事是确定的：我们为确定性系统磨了二十年的那套工具链，在面对 Agent 时集体找不着北。不确定的是——替代它的是什么。

这个问题可能不会在技术层面获得一个干净的答案。因为确定性本身就不是 AI 的特性。如果你一定要追求确定，那你可能从一开始就不应该用 Agent。
