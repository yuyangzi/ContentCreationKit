# 审核主题

## 目标

对已选定的创作主题进行深度审核，确保方向明确、细节清晰、背景资料充分。

## 前置条件

已完成 `/find-popular-topics`，或已有确认的创作主题

## 输入

- `content/topics/` 目录下的主题文件，或对话中已确认的主题

## 步骤

1. 使用 `recursive-research` skill 和搜索类 MCP 工具，深度收集与主题相关的背景信息。
2. 使用 `grill-me` skill 结合背景信息对主题内容进行反复拷问和细节确认，一轮只问一个关键问题，直到没有任何模糊点。
3. 如果发现有需要补全的信息同样使用 `recursive-research` skill 和搜索类 MCP 工具进行搜索。
4. 将确认的问答记录和收集的信息整理到 `content/reference/` 目录，文件名为 `{时间戳}-{主题名}.md`。
5. 给出修正意见供我审核，等待用户确认。
6. （可选项）根据用户确认的修正意见，询问是否对 topic 文件执行结构性操作：

   | 操作 | 触发条件 | 执行方式 |
   |------|----------|----------|
   | **合并** | 多个 topic 内容高度重叠 | 合并为一个文件，旧文件执行 `git rm` |
   | **拆分为多个** | 单一 topic 跨度过大、可拆为独立主题 | 拆分为多个文件，原文件标注「已拆分」 |
   | **更新** | 审查发现事实错误或需要补充 | 追加 `## {YYYY-MM-DD} 更新` 区块，同步更新文件名时间戳 |
   | **调整标题/标签** | 方向偏移或分类错误 | 修改文件内元数据 |

   - 如果用户选择跳过，不修改任何 topic 文件
   - 操作前先列出变更清单供确认

## 输出

- `content/reference/{时间戳}-{主题名}.md`：包含审核问答记录和背景研究资料
- （可选）更新后的 `content/topics/` 文件

## 约束

**必须遵守：**

- 一条一条提问，每轮只问一个关键问题
- 引用信息须提供来源，不得捏造
- 步骤6 的操作须在用户确认后方可执行

**禁止操作：**

- 不要一次性抛出多个问题
