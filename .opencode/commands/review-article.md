# 文章内容审查

## 目标

模仿 `review-draft` 的逐条检查结构，对文章进行 **6 个 StyleRule 维度** + **1 个 deep-tech 专项** + **1 个综合段**共 8 步检查，产出可执行的修改清单。

数据准确性只是其中一个维度，还要检查：论证逻辑、技术对比公平性、哲学升华空洞性、局限性诚实度、引用一致性、术语精确性。

## 模式判断

读取文章 frontmatter 的 `mode` 字段：

- `mode = deep-tech` → 启用 **deep-tech 专项检查（步骤 7）** + **3 个并行 librarian 验证**
- `mode = standard` 或缺失 → 跳过步骤 7，单 librarian 验证

## 前置条件

已完成 `/to-article`，`content/article/` 目录已有正式文章

## 输入

- `content/article/` 目录下的文章文件（含 frontmatter 的 `mode` 字段）
- 额外审核意见（如有）
- **【conservative 模式】** `content/article/{YYYYMMDD}-{主题名}-changes.md` 润色改动清单（如有）

## 加载 Skill

**deep-tech 模式**：并行 3 个 librarian（team_mode）：

| Librarian | 焦点 | 验证内容 |
|-----------|------|----------|
| librarian-1 | 架构机制准确性 | arXiv 编号、技术细节、对比表数据 |
| librarian-2 | 商业数据时效性 | 价格、融资、估值、市值、参数量 |
| librarian-3 | 叙事张力 | 逻辑、过渡、哲学升华、局限性诚实 |

**standard 模式**：单 librarian 或 2 个 librarian（与现状一致）。

## 步骤

### 1. 作者隐身（继承 StyleRule §1）

逐项检查：
- [ ] **元叙述**：没有"写到这里"、"我注意到"、"讲到这里就绕不开"等
- [ ] **剧情预告**：没有"接下来你会看到"、"更精彩的还在后面"等
- [ ] **认知反转展示**：没有"一开始我也是这么想的，但查完资料发现..."等
- [ ] **设问修辞**：没有替读者提问再自答（"讲到这里就绕不开一个问题...答案要从 XX 说起"）
- [ ] **情绪标签**：没有"令人意外的是"、"有趣的是"、"讽刺的是"等
- [ ] **密集"我"**：没有连续出现的"我也"、"我发现"、"我注意到"

**判断标准**：删掉这句话，读者还能不能跟上叙事？能就删。

### 2. 弱化绝对（继承 StyleRule §2）

逐项检查：
- [ ] **绝对断言**：没有"全都"、"从来不"、"不管 XX 都不会"等
- [ ] **未来预测**：有对未来判断时加了限定（"至少当前"、"目前来看"）
- [ ] **过度概括**：没有把个别案例包装为普遍规律
- [ ] **deep-tech 模式特别检查**：没有"全面升级"、"完美解决"、"彻底击败"等绝对词

### 3. 论据收敛（继承 StyleRule §3）

逐项检查：
- [ ] **程度重申**：对比已成立，没有补"整整早了 X 时间"类冗余
- [ ] **数据堆砌**：一组数据足以支撑判断后，没有追补更多数据
- [ ] **多例枚举**：同一论点没有用两三个并列例子反复论证
- [ ] **背景铺垫**：引用数据前没有花半段介绍公司背景
- [ ] **防杠声明**：没有"截至 X 时间"这类不影响论点的时效声明
- [ ] **来源标注**：关键数据首次出现括号附注来源，后文不重复标注

### 4. 叙事纯净（继承 StyleRule §4）

逐项检查：
- [ ] **新闻聚合段**：没有"有人这么说、有人那么说"的综述段
- [ ] **背景推论岔路**：主线论证不是建立在"事实如此"上的推论岔路
- [ ] **对比案例岔路**：论证 A 时没有"顺便提一下 B"的对比岔路
- [ ] **戏剧化过渡**：没有"如果唯速度论，企业可能会栽到另一个坑里"类预警
- [ ] **过渡句纪律**：过渡只做方向转折（"但/然而"）或逻辑推进（"更进一步/这意味着"），不附加情绪悬念

### 5. 列表化（继承 StyleRule §5）

逐项检查：
- [ ] **并列结构**：3+ 同级并列信息已用列表，而非装在散文段落中
- [ ] **适度使用**：没有追求"全篇列表化"——只在需要一次性接收多个并列信息时才用
- [ ] **排版规范**：列表前后各加空行

### 6. 破折号约束（继承 StyleRule §6）

逐项检查：
- [ ] **单段破折号数**：单段 `——XX——` 不超过 1 处
- [ ] **括号替代**：能用 `(XX)` 安静补充时不强行用破折号
- [ ] **断句重写**：超过 1 处的句子改为重新断句或拆分

### 7. deep-tech 专项检查（deep-tech 模式强制）

仅当 `mode = deep-tech` 时执行，3 个并行 librarian 协同验证：

- [ ] **arXiv 编号准确性**：每个 arXiv 编号可在 arxiv.org 检索到，标题匹配
- [ ] **技术对比公平性**：横向对比不抬高自家贬低对手，数据来源一致
- [ ] **哲学升华空洞性**：升华段不空喊口号（避免"开启了新时代"类），有具体技术判断
- [ ] **局限性诚实度**：显式承认短板，不回避核心争议（如 K3 幻觉率 51% 这种数据必须显式提及）
- [ ] **术语精确性**：首次出现的术语中文翻译准确，英文原形保留
- [ ] **参考来源完整性**：文末 `*参考来源*` 段，链接有效，发布方明确
- [ ] **数据/基准可追溯**：关键数字首次出现标源，后文不重复

### 8. 综合检查

- [ ] 文章的数据和内容真实且未过时
- [ ] 上下文结构合理，前后逻辑正确，语句通顺
- [ ] **【conservative 模式】** 润色改动清单与文章实际改动一致

## 输出

按 8 个检查节顺序呈现，每项附"**通过 / 需修改 / N/A**"。需修改项必须给：
- 具体位置（行号或原文引用）
- 例证（引用问题原文）
- 建议改法

最终附 1 段"**修改优先级清单**"，按三档分类：

| 优先级 | 含义 |
|--------|------|
| **必须改** | 影响准确性 / 严重违反 StyleRule / 数据错误 |
| **建议改** | 影响可读性 / 局部违反 StyleRule |
| **可选改** | 风格优化 / 措辞微调 |

## 约束

**必须遵守：**

- 8 个检查节按顺序逐项检查，不能跳过或笼统带过
- 每项检查需给出具体问题位置（引用原文），不笼统说"已通过"
- **deep-tech 模式**：必须完成步骤 7 的所有 7 个子项
- **deep-tech 模式**：3 个 librarian 并行验证后必须综合分析
- **conservative 模式**：核对 `changes.md` 与文章实际改动是否一致

**禁止操作：**

- 不要直接修改文章内容
- 不要未经确认直接修改文章
- 不要只给笼统评价，必须指出具体问题位置
- 不要笼统说"已通过"，必须给具体判断
- 不要仅凭模型训练数据判断真伪，须以搜索查证结果为准
