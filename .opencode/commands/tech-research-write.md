# 技术知识点深度研究与写作

## 目标

用户给出一个具体的技术知识点（如 "Transformer 的 KV Cache 原理"、"HyDE 假设性文档嵌入"、"MoE 路由机制"），命令自动完成三步流程：

1. 深度研究该知识点的核心原理、最佳实践和最新进展
2. 使用 grill-me 拷问模糊点、创作方向和文章结构
3. 生成一篇讲解分析型技术文章草稿

## 前置条件

（无）

## 输入

- 用户指定的技术知识点（一句话描述即可，如 "RAG 中的 HyDE 技术"）
- （可选）用户补充的创作方向、目标读者、篇幅偏好

## 加载 Skill（强制）

| 阶段 | 加载 Skill |
|------|-----------|
| 阶段一（信息收集） | 无需加载（使用 librarian agent + web search） |
| 阶段二（grill-me 拷问） | `grill-me` |
| 阶段三（生成草稿） | `writer-style`（技术深读模式）+ `humanizer` + `content-research-writer` |

## 步骤

### 阶段一：信息收集

1. **确认知识点**：向用户确认要研究的技术知识点，明确范围边界（如 "只讲 KV Cache 原理，不讲 FlashAttention"）。如果用户未指定边界，主动询问范围。

2. **派遣 librarian agent 并行搜索**（至少 3 个，**同时发起**，`run_in_background=true`）：
   - Agent 1 — 核心原理：搜索论文原文（arXiv）、官方文档、权威教程，提取核心概念与工作机制
   - Agent 2 — 最佳实践：搜索工程落地经验、常见误区、性能优化技巧、GitHub 上的实际使用案例
   - Agent 3 — 最新进展：搜索该知识点的最新变体、相关改进、社区讨论、替代方案对比

3. **补充 web search**：对 librarian 未覆盖的盲区，使用 Tavily / BingSearch / ExaSearch 补搜。优先搜索中文社区（知乎、掘金、InfoQ）对应该知识点的讨论，了解国内读者的认知基础和常见困惑。

4. **整理研究摘要**：将所有收集到的信息整合为结构化摘要，在对话中呈现给用户。摘要包含：
   - **核心概念**：一句话定义 + 工作机制简述（含公式/伪代码，如适用）
   - **关键来源**：论文（arXiv 编号）、官方文档、权威博客（链接 + 发布日期）
   - **最佳实践**：3-5 条可操作的工程建议
   - **常见误区**：2-3 个典型理解偏差或踩坑记录
   - **最新进展**：近 6 个月的重要更新或变体
   - **代码示例**：GitHub 仓库链接（如有高质量开源实现）

### 阶段二：grill-me 拷问

5. **加载 `grill-me` skill**，基于阶段一的研究摘要，对用户进行深度拷问。**至少 3 轮，每轮只问一个问题**。典型问题方向（按优先级排序）：

   | 优先级 | 拷问方向 | 示例问题 |
   |--------|----------|----------|
   | 1 | **模糊点清除** | "研究摘要里提到 XX 机制有两种理解方式——你认为哪种更接近实际？还是你想两者都写？" |
   | 2 | **创作方向** | "这篇文章的目标读者是谁？是刚入门需要科普的，还是已经用过想深入理解的？" |
   | 3 | **文章结构** | "你倾向哪种叙事结构——问题驱动（先抛痛点）、历史演进（从旧方案到新方案）、还是对比分析（A vs B）？" |
   | 4 | **取舍决策** | "这个知识点涉及 A、B、C 三个子话题，篇幅有限，哪些必须展开、哪些一笔带过？" |
   | 5 | **独特视角** | "在 XX 的实际使用中，你有没有踩过坑或有个人经验可以融入文章？" |

6. 记录所有 grill-me 问答，作为阶段三草稿生成的输入。

### 阶段三：生成草稿

7. **加载写作 skill**：`writer-style`（技术深读模式）+ `humanizer` + `content-research-writer`。

8. **生成草稿**：基于研究摘要 + grill-me 问答结果，生成技术讲解文章草稿。草稿必须包含以下要素：

   - [ ] **概念解释**：术语首次出现时括号注释（如 "KV Cache（键值缓存，推理时缓存已计算的 Key 和 Value 矩阵）"）
   - [ ] **代码示例**：至少 1 个精简的代码片段（Python/伪代码），演示核心逻辑
   - [ ] **最佳实践**：独立小节总结 3-5 条工程建议
   - [ ] **常见误区**：指出 2-3 个典型理解偏差
   - [ ] **来源引用**：关键数据/观点附来源（论文 arXiv 编号、官方文档链接）
   - [ ] **StyleRule 合规**：符合 `StyleRule.md` 六条规则（作者隐身、弱化绝对、论据收敛、叙事纯净、列表化、破折号约束）

9. 以 `{YYYYMMDD}-{知识点主题名}.md` 的文件名保存到 `content/draft/` 目录。

## 输出

- `content/draft/{YYYYMMDD}-{知识点主题名}.md`：完整的技术讲解文章草稿
- 研究摘要（在对话中呈现）
- grill-me 问答记录（在对话中呈现）

## 与主创作管线的关系

本命令是独立于主创作管线（`/find-popular-topics` → `/to-wechat`）的快捷入口，适用于：

- 用户已有明确想写的技术知识点，不需要从热点选题开始
- 想快速完成"研究 → 草稿"的闭环

草稿生成后，可衔接主管线：

```
/tech-research-write  →  /review-draft  →  /to-article  →  /review-article  →  /to-wechat
```

## 约束

**必须遵守：**

- 阶段一的 librarian agent 必须并行派遣，不能串行（`run_in_background=true`）
- 阶段二的 grill-me 至少 3 轮，每轮只问一个问题
- 所有技术数据必须可追溯到原始来源（论文、官方文档、权威博客）
- 草稿必须符合 `StyleRule.md` 六条规则
- 不使用未核实的数据和引用
- 不使用 AI 腔（空洞的形容词堆砌、"首先/其次/最后/综上所述"模板化连接词）

**禁止操作：**

- 不要跳过 grill-me 直接生成草稿
- 不要编造数据、论文、代码示例
- 不要把"术语解释"写成百科词条风格
- 不要在没有用户确认范围边界的情况下开始研究
