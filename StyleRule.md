# 编辑风格规则

基于 `20260617-微信AI专属卡-支付宝阿宝-超级App-Agent化.md` 的修改 diff 归纳。

---

## 1. 砍元叙述（Meta-narrative）

**核心原则**：不要让读者看到作者在"组织内容"的过程。

删除以下类型的句子：

- "写到这里我想起一件事"、"写到这里，我想把 XX 再往前推一步" → 直接推进
- "我注意到一个细节：" → 直接说细节
- "我反复看了几遍" → 不需要交代观看次数
- "最直接的方式是看产品形态" → 直接看
- "注意这个细节：" → 直接说细节
- "一开始我也是这么想的。但仔细查询完资料后，我发现事情并不是这样" → 直接给结论"我并不是这么认为"
- "讲到这里就绕不开一个问题……答案要从 XX 说起" → 直接说 XX

**更深一层**：不展示"我如何得出结论"的完整过程（查资料→比对→发现→反转），而是直接给出最终判断。读者不需要看你走过的弯路。

接受"划重点"这类简短直接的引导词，但不接受作者站出来描述自己的写作行为或思维过程。

## 2. 弱化绝对化断言，增加限定词

**核心原则**：对"斩钉截铁"的判断有本能反感。

- 将绝对断言改为有边界的判断
  - ❌ "不管未来 AI 进化到什么程度，XX 都不会被吞进去"
  - ✅ "至少在当前的产品架构设计中，AI 不会入侵资产"
- 用"大多数""通常""倾向于"替代"全都""从来不""出奇一致"
- 不对未来做武断预测，如果必须要说，加限定条件（"至少""当前""目前来看"）

不是立场不坚定，而是认为"绝对正确"本身就不正确。

## 3. 砍冗余说明

**核心原则**：如果论点本身就足以说明问题，论据点到为止，不要反复强调。

删除以下类型的冗余：

- **已经说了 A 比 B 早，就不需要再说"整整早了 X 时间"**。对比本身已经成立，时间和程度数字是多余的重申。
- **已经说了"这件事是媒体披露的"，就不需要再说"时间还原回去之后会发现一个细节"**。框架从句不增加信息量。
- **已经说了"原文引用"，就不需要再说"有一句原话，我反复看了几遍"**。看几遍不重要，原话本身才重要。
- **合并重复信息段落**。两段话说的是一件事就合成一段，不拆开装成两个发现。

追求论点效率，不是论据密度。一个论据够了就不需要第二个。

## 4. 砍"过于具体"的例子

**核心原则**：例子的作用是支撑论点，论点本身成立时例子点到为止。

- 用列表中的最后一项举例 → 可以删。前面几个已经说明问题了。
- 用某个具体品牌细化场景 → 可以删。读者能懂场景，不需要具体名字。
- 用两三个并列例子枚举 → 留一个就够了。

判断标准：去掉这个例子，论点是否仍然清晰成立？如果是，删。

## 5. 结构化信息用列表，非散文段落

**核心原则**：当信息是多个并列要点时，用列表结构，不让读者在段落里自己找要点。

- 3 个及以上并列特征/原因/维度 → 用 `1. 2. 3.` 或 `- ` 列表
- 每个列表项内部可以是一句到几句，但列表项之间是同级并列关系
- 列表前后各加空行

不追求"全篇列表化"，只在"读者需要一次性接收多个并列信息"时才用。

## 6. 第一人称在场但不出头

**核心原则**：接受"我"作为叙述者，但"我"是叙事底色，不是前台焦点。

- "我一开始也是这么想的" → ❌ 实际上删掉了。不展示"我是怎么从 X 变成 Y 的"，直接说 Y
- "我注意到一个细节" → ❌ "我"太抢戏，直接说细节
- "我已经回答过了" → ✅ 可用，但改成"所以说 XX 是不是……我已经回答过了"（用过渡弱化"我"的存在感）
- "我仔细查询完资料后，我发现事情并不是这样" → ❌ 删掉"我发现"这个过程，直接给结论
- 删除"我"的重复使用——"我也""我发现""我注意到"连续出现时，"我"就太多了

让读者感受到有人在说，但不让那个人跳出来挥手。

## 7. 数据服务于论点节奏，不堆砌

**核心原则**：数据完成了逻辑支撑的任务就收住，不要继续堆。

- 一组时间线 + 一组对比数据足以支撑判断后，不要接着补更多数据做"证明"
- 砍掉以下模式："具体来说，XX 在 Y 时间达到了 Z 数据；而 AA 同期只有 BB……因此 CC 出现了 DD 的变化"。如果读者已经能推出结论，中间过程可以整段压缩为一句话
- 判断标准：删掉这段数据，论点是否仍然站得住？如果论点本身已经通过前文逻辑建立，数据可以砍

不是不用数据，而是不让数据喧宾夺主。好的论证节奏是"逻辑推进→数据点到为止→继续推进"，不是"逻辑→数据→更多数据→更多数据→结论"。

## 8. 标题去术语化，去行业黑话

**核心原则**：标题用读者能一眼看懂的话，不用行业术语包装深度。

- "三方卡位：表面是抢入口，底下是范式威胁" → ❌ "卡位""范式威胁"都是行业黑话
- "全新的交互范式，颠覆的用户习惯" → ✅ 直白，读者不用翻译
- "产品级、协议级、生态级三层换芯" → ❌ 听着像分析师报告
- "小程序变 Skill：协议层的预谋" → ✅ "预谋"口语化，有画面感

判断标准：标题拿掉之后，给一个不懂行业术语的人看，ta 能猜出大概在讲什么。

## 9. 删设问修辞，不替读者提问

**核心原则**：不要用"讲到这里就绕不开一个问题……"这种设问来自答推进文章。

- "讲到这里就绕不开一个问题：既然不是反击豆包，那腾讯阿里在和谁较劲？" → ❌ 设问腔，作者替读者提问
- "引入另一个概念来解释" → ❌ 同样是设问变体
- 直接推进论点，不需要假想一个读者的问题来承接

设问与元叙述的区别：元叙述是作者在说"我在写作"，设问是作者在说"你应该会问"。两种都是多余的表演。

## 总结：判断优先级

修改时按这个顺序决策：

1. **这句话是不是说得太绝对了？** → 加限定
2. **这句话有没有增加新信息？** → 没增加就删
3. **这个例子是不是必须的？** → 不是就删
4. **这几个要点能不能变列表？** → 能就改
5. **"我"在这里是不是太显眼了？** → 是就调暗
6. **这段时间线是不是论证必需的？** → 不是就整段删
7. **这堆数据是不是堆上去的？** → 是就压缩
8. **这个标题是不是行业黑话？** → 是就改白话
9. **这句话是不是设问修辞？** → 是就删
